# 下一阶段开发方案（M2-M4）

## 1. 背景与目标

当前项目已完成 M1（爬虫主链路）能力建设，具备 Playwright 渲染、登录检测、滚动采集、详情解析与幂等入库能力。下一阶段目标是把“可运行”升级为“可观测、可诊断、可稳定运营”。

本方案聚焦三个方向：
- M2：可视化预览与诊断链路
- M3：稳定性与异常治理
- M4：部署运行与运维保障

## 2. 范围定义

### 2.1 In Scope
- 后端：任务预览字段、预览查询接口、状态与错误信息链路
- 爬虫：关键步骤状态上报、失败原因标准化、预览快照产物上报
- 前端：任务列表增强、预览面板、异常可视化、轮询策略优化
- 运维：镜像固化、容器启动一致性、基础监控与日志排查手册

### 2.2 Out of Scope
- 大规模分布式调度架构重构
- 多站点采集策略统一平台化
- 高级算法级反爬对抗方案

## 3. 总体实施策略

- 采用“先可观测，再提稳定，最后做运维收敛”的分层推进方式。
- 每个阶段都要求“实现 + 验证 + 文档更新”闭环，避免文档与代码漂移。
- 全量验证统一在 Docker 环境执行，确保与部署环境一致。

## 4. 分阶段方案

### 4.1 M2：可视化预览与诊断链路

### 目标
- 用户创建采集任务后，能持续看到“进度、最新步骤、错误原因、最近预览”。
- 排查时可通过任务详情快速定位“卡在哪一步、为何失败”。

### 关键改造
- 数据模型扩展
  - `CrawlTask` 增加预览与诊断字段：
    - `latest_step`
    - `error_message`
    - `latest_frame_url`
    - `preview_updated_at`
- 后端接口
  - 增加任务预览查询接口（按任务 ID 返回当前预览状态）
  - 扩展任务列表字段，支持前端直接展示状态摘要
- 爬虫状态上报
  - 在关键步骤更新 `latest_step`
  - 失败时写入 `error_message`
  - 生成或更新预览产物后写入 `latest_frame_url/preview_updated_at`
- 前端展示
  - Dashboard 增加“任务预览”区域（最近一帧、步骤、错误）
  - 状态标签与错误信息映射统一，避免前后端语义不一致

### 验收标准
- 新建任务后，列表中可实时看到步骤变化
- 任务失败时可见明确错误信息，而非通用失败
- 至少一条任务可展示预览信息并可刷新

### 4.2 M3：稳定性与异常治理

### 目标
- 降低“任务触发成功但有效数据为 0”的不可见风险。
- 建立标准化异常分类与重试边界，避免隐性失败。

### 关键改造
- 异常分类
  - 网络异常、反爬拦截、页面结构异常、数据清洗异常分层标记
- 重试与超时策略
  - 明确请求级重试、步骤级重试与任务级终止条件
  - 为关键步骤设置超时并上报超时原因
- 数据质量守护
  - 增加最小有效结果阈值告警（如抓取 0 条时标记为可疑成功）
  - 补充关键字段空值监控（title/company/source_url 等）
- 可观测性增强
  - 统一任务日志字段（task_id、step、error_type）
  - 保证前端可读取到必要诊断信息

### 验收标准
- 常见异常均可定位到明确错误类型
- “0 结果成功”可被识别并标记为需关注任务
- 相同输入下任务结果稳定性明显提升（以错误率下降为依据）

### 4.3 M4：部署与运维保障

### 目标
- 保证构建、启动、验证流程可重复执行，降低环境差异影响。
- 建立面向日常维护的标准操作文档。

### 关键改造
- 镜像与容器
  - 持续维护 crawler 镜像依赖固化（Playwright 运行库与浏览器）
  - 保留基础镜像参数化能力，兼容网络受限构建场景
- 运行策略
  - 明确容器启动顺序与健康检查策略
  - 明确日志查看、失败重启、资源清理策略
- 运维文档
  - 新增“构建失败排查”、“采集无数据排查”、“预览异常排查”手册
  - 将常见故障处理沉淀为 checklist

### 验收标准
- 新环境可按文档完成构建、启动、基础验证
- 常见故障有可执行排查路径，不依赖口头经验

## 5. 任务拆解与交付物

### M2 交付物
- 后端字段迁移与序列化更新
- 预览接口与任务列表字段扩展
- 前端预览面板与状态映射
- M2 验证报告与接口说明文档

### M3 交付物
- 异常分类规范文档
- 重试/超时策略配置化改造
- 数据质量守护规则
- 稳定性验证记录

### M4 交付物
- 镜像固化规范与构建参数说明
- 运维排查手册与故障 checklist
- Docker 环境回归验证记录

## 6. 风险与应对

- 外部站点反爬策略变化
  - 应对：异常分类 + 预警 + 规则可调
- Docker Hub 或外网不稳定
  - 应对：基础镜像参数化 + 本地镜像兜底构建路径
- 文档与实现偏离
  - 应对：每阶段强制同步更新 docs 与 specs
- 预览资源增长过快
  - 应对：预览保留策略与清理任务

## 7. 验证策略

- 单项验证
  - 接口响应字段完整性
  - 前端展示正确性
  - 爬虫步骤/错误上报有效性
- 集成验证
  - “创建任务 -> 执行 -> 预览展示 -> 结果入库”全链路
- 回归验证
  - 对 M1 现有能力做最小回归，避免新改造破坏主链路
- 环境约束
  - 所有验证在 Docker 容器环境执行

## 8. 文档与规范更新要求

- 每完成一个阶段，必须同步更新：
  - `docs/文件级开发任务拆解清单.md`
  - `docs/开发问题复盘.md`
  - `.trae/specs` 对应阶段文档
- 对外说明统一使用“现象 -> 根因 -> 修复 -> 验证”结构。

## 9. 下一步执行入口

- 优先启动 M2：
  - 先完成后端字段与接口，再落前端预览面板，最后联调验证。
- M2 完成后进入 M3：
  - 以“0 结果成功”问题为第一优先级建立稳定性治理闭环。
- M3 稳定后执行 M4：
  - 固化构建与运维流程，形成可复用交付标准。
