## 一、登录问题修复复盘

### 问题 1：未登录状态访问业务页，体验不稳定

- **现象**
  - 访问首页或职位页时，未登录用户可能无法立即感知需要登录。
- **根因**
  - 业务路由需要统一基于登录状态做前置守卫与跳转策略。
- **修复**
  - 在路由层增加鉴权重定向逻辑：受保护路由未登录时跳转 `/login`，并保留 `redirect` 参数；已登录访问 `/login` 时回首页。
  - 对应实现：`frontend/src/router/index.js` 中 `resolveAuthRedirect` 和 `beforeEach`。
- **验证**
  - 未登录直达 `/` 或 `/jobs`，会被跳转到 `/login`。
  - 登录后会回跳到原始目标路由。

### 问题 2：登录状态在刷新后容易丢失

- **现象**
  - 页面刷新后，前端状态可能回到初始值，导致再次被判定未登录。
- **根因**
  - 登录状态如果只在内存中保存，刷新后无法恢复。
- **修复**
  - 在认证 Store 中将 `accessToken/refreshToken` 持久化到 `localStorage`，并在初始化时回填。
  - 对应实现：`frontend/src/store/auth.js`。
- **验证**
  - 登录后刷新页面，仍可访问受保护页面。

### 问题 3：登录失败提示可用性不足

- **现象**
  - 用户登录失败时很难判断是账号密码错误还是服务不可用。
- **根因**
  - 异常提示信息过于笼统，未优先展示后端返回消息。
- **修复**
  - 登录失败时优先读取后端 `message`，否则回退通用提示。
  - 对应实现：`frontend/src/views/Login.vue` 的 `onSubmit` 异常处理。
- **验证**
  - 后端返回明确报错时，页面可直接提示具体失败原因。

## 二、本次采集“无动静”修复复盘

### 问题 1：点击“开始采集”后用户看不到过程反馈

- **现象**
  - 点击按钮后仅出现“已创建采集任务”，页面缺少任务执行过程与结果展示，主观感受为“没有反应”。
- **根因**
  - 前端只调用创建任务接口，没有任务列表展示，也没有状态轮询。
  - 采集是 Celery 异步执行，创建成功不等于采集完成。
- **修复**
  - 新增任务列表接口调用：
    - `frontend/src/api/index.js` 增加 `fetchCrawlTasks`。
  - Dashboard 新增“采集任务状态”面板与状态标签：
    - `frontend/src/views/Dashboard.vue` 增加任务表格、状态映射、计数展示。
  - 新增自动轮询：
    - 当任务为 `pending/running` 时，按间隔刷新任务列表；
    - 任务全部结束时停止轮询；组件卸载时清理定时器。
- **验证**
  - 创建任务后可看到任务行即时出现；
  - 状态会在“排队中/采集中/已完成/失败”之间变化；
  - 成功/失败计数会更新。

### 问题 2：本地直接构建受 PowerShell 执行策略影响

- **现象**
  - 在 Windows 本地终端执行 `npm run build` 报脚本执行策略错误。
- **根因**
  - 本地 PowerShell 执行策略限制 `npm.ps1`。
- **修复**
  - 使用容器内命令进行构建验证，避免宿主机策略差异干扰：
    - `docker compose exec frontend npm run build`
- **验证**
  - 容器内构建通过。

### 问题 3：任务已执行但抓取结果为 0（非本次 UI 反馈问题）

- **现象**
  - Celery 日志显示任务执行成功，但 `total_count` 为 0。
- **根因**
  - 说明“任务触发链路”正常，但爬虫侧数据抓取结果为空，常见原因包括目标站反爬、页面结构变化、请求参数不匹配等。
- **处理结论**
  - 本次已先修复“无反馈”问题；
  - 抓取结果为 0 属于下一阶段“爬虫有效数据率”问题，需单独排查 spider 与目标站兼容性。

- 验证优先使用与部署一致的运行环境：

## 五、本次“环境固化到镜像”复盘

### 问题 1：直接 `docker compose build crawler` 构建失败

- **现象**
  - 构建阶段拉取 `python:3.11-slim` 失败，报 Docker Hub 连接超时。
- **根因**
  - 当前环境到 `registry-1.docker.io` 网络不稳定，基础镜像元数据获取失败。
- **修复**
  - Dockerfile 增加 `ARG BASE_IMAGE`，默认仍为 `python:3.11-slim`，并支持在网络受限时切换到本地已存在镜像。
  - 使用本地已有 `boss-job-platform-crawler:latest` 作为基底完成一次增量重建，确保固化流程可落地。
- **验证**
  - 镜像成功构建并替换运行中的 crawler 容器。

### 问题 2：容器内缺少 `scrapy-playwright` 导致启动报错

- **现象**
  - 爬虫启动时报 `ModuleNotFoundError: No module named 'scrapy_playwright'`。
- **根因**
  - 运行容器基于旧镜像，镜像内依赖未更新到最新 requirements。
- **修复**
  - 先在容器内临时安装依赖恢复可用性；
  - 再将依赖安装步骤固化到 Dockerfile，并重建镜像保证后续环境一致。
- **验证**
  - 容器内 `scrapy check` 可通过，`scrapy crawl` 可启动到 Playwright 阶段。

### 问题 3：Playwright 配置路径不兼容

- **现象**
  - 启动时报 `No module named 'scrapy_playwright.middleware'`。
- **根因**
  - `scrapy-playwright` 新版本不再提供该 middleware 路径，错误配置导致中间件加载失败。
- **修复**
  - 移除 `DOWNLOADER_MIDDLEWARES` 中无效项，仅保留 `DOWNLOAD_HANDLERS` 的 Playwright 处理器配置。
- **验证**
  - Scrapy 可正常加载下载器并启动 spider。

### 问题 4：浏览器二进制与系统依赖不完整

- **现象**
  - 报错 `Executable doesn't exist ... headless_shell`，或提示缺少 `libglib/libnss/libx11` 等运行库。
- **根因**
  - 镜像中未完整安装 Playwright 运行依赖与 headless shell 浏览器。
- **修复**
  - 在 Dockerfile 固化安装系统库与字体；
  - 固化 `playwright install --only-shell chromium`；
  - 固化 `PLAYWRIGHT_BROWSERS_PATH=/ms-playwright`，确保浏览器路径可预测。
- **验证**
  - 容器内执行 Playwright 启动脚本输出 `playwright_ok`；
  - 采集命令日志出现 `Browser chromium launched`。

### 问题 5：容器内 apt 安装偶发锁冲突

- **现象**
  - 安装依赖时出现 `Could not get lock /var/lib/dpkg/lock-frontend`。
- **根因**
  - 前一次 apt 进程未完全释放锁文件，导致后续安装失败。
- **修复**
  - 清理锁并执行 `dpkg --configure -a` 后重试安装流程。
- **验证**
  - 系统库安装成功，后续 Playwright 安装和启动均正常。

## 七、本次智联（zhaopin.com）适配改造总结

### 1) 修改目的

- 明确当前主抓取站点为 `zhaopin.com`，避免混入非智联域名页面。
- 基于最新智联页面结构修正登录态判断，减少“误判已登录/未登录”。
- 收紧详情页链接识别规则，避免将公司页、活动页误当职位详情页解析。
- 在 Docker 运行环境完成闭环验证，保证与部署环境一致。

### 2) 改动情况

- 登录态判断增强（对齐新页面 HTML）：
  - 新增用户态选择器：`.home-header__c-login`、`.c-login__top__name`、`.home-header__avatar`、`a[href*='//i.zhaopin.com']` 等。
  - 新增未登录态选择器：`.home-search__c-no-login`、`.home-header__a-login`、`.home-header__b-login`。
  - 文件：`crawler/crawler/spiders/zhaopin_jobs.py`
- 域名过滤增强：
  - 新增 `_is_zhaopin_url`，仅允许 `zhaopin.com` 及其子域名。
  - 在 `_normalize_url`、`_normalize_detail_urls` 中统一使用该校验。
  - 文件：`crawler/crawler/spiders/zhaopin_jobs.py`
- 详情链接识别增强：
  - 新增 `_is_job_detail_url`，仅保留职位详情相关 URL 模式（`/job/`、`jobs.zhaopin.com`、`sou.zhaopin.com/jobs`）。
  - 在“初始数据抽取”和“滚动抓链路”两处同时启用。
  - 文件：`crawler/crawler/spiders/zhaopin_jobs.py`
- 登录态校验脚本增强：
  - `scripts/check_zhaopin_auth.py` 增加域名判定 `domain_ok`，并将 `usable` 收紧为 `logged_in && domain_ok`。

### 3) 遇到的问题

#### 问题 A：校验脚本初次运行失败

- **现象**
  - 本机执行 `python scripts/check_zhaopin_auth.py` 报 `ModuleNotFoundError: No module named 'playwright'`。
- **根因**
  - 当前本机 Python 环境未安装 Playwright 依赖。
- **解决**
  - 先完成脚本语法与逻辑修正，再将功能验证切换到 Docker 容器环境执行。

#### 问题 B：详情链接混入非职位页面

- **现象**
  - 抓取结果中出现 `companydetail` 等公司主页类型链接，字段质量不稳定。
- **根因**
  - 旧逻辑对“智联域名”过滤较宽，缺少“职位详情 URL”强约束。
- **解决**
  - 增加 `_is_job_detail_url` 并在两条链接采集路径都做拦截，仅保留职位详情 URL。

#### 问题 C：缺少部署一致性验证

- **现象**
  - 本机直接验证通过不能完全代表容器实际运行情况。
- **根因**
  - 本机与容器在依赖、运行时、浏览器环境上存在差异。
- **解决**
  - 在 Docker 中执行编译与实跑验证，作为最终验收依据。

### 4) 解决结果与验证

- Docker 编译校验通过：
  - `docker compose run --rm crawler python -m py_compile crawler/spiders/zhaopin_jobs.py`
- Docker 实跑通过（带登录态）：
  - `docker compose run --rm -e ZHAOPIN_STORAGE_STATE=/app/crawler/auth_zhaopin.json crawler scrapy crawl zhaopin_jobs -a keyword=python -a max_pages=1 -a crawl_task_id=docker_test -s CLOSESPIDER_TIMEOUT=60 -s LOG_LEVEL=INFO`
- 关键结果：
  - `response_count = 10`
  - `item_scraped_count = 7`
  - `crawl_task_id = docker_test`
  - 未再出现本轮前的 `companydetail` 误入主抓链路现象

## 八、今日开发问题总结（

### 问题 1：详情页字段匹配对页面结构过于敏感

- **现象**
  - 经验/学历/地点等字段在不同模板页出现错位或误判，发布时间文本带前缀噪声。
- **根因**
  - 旧规则依赖固定节点顺序（`nth-child`）和单一路径，缺少语义回退与统一清洗。
- **修复**
  - 在 `zhaopin_jobs.py` 中增加文本清洗、列表去重、语义识别回退（经验/学历/地点）和发布时间标准化。
  - 扩展职位详情 URL 识别规则，减少漏抓与错抓。
- **验证**
  - Docker 内 `python -m py_compile crawler/spiders/zhaopin_jobs.py` 通过。
  - Docker 内规则冒烟断言通过（经验/学历/地点/发布时间/详情链接均符合预期）。

### 问题 2：点击“开始采集”后反复提示创建失败

- **现象**
  - 前端点击“开始采集”后弹出“创建失败，请稍后重试”。
  - 同时在前端容器日志可见 Vite 代理报错：`http proxy error` / `ECONNREFUSED`。
- **根因**
  - 前端请求路径是 `POST /api/crawl/tasks`，通过 Vite 代理转发到 backend。
  - 在容器启动窗口期，frontend 先可用但 backend 尚未就绪，代理连接 backend 失败，前端进入兜底报错分支。
  - 该兜底文案仅表示“未拿到可解析错误体”，并不等价于后端业务校验失败。
- **修复**
  - Docker 编排侧增加健康检查与启动顺序约束，避免 frontend 早于 backend 就绪：
    - `postgres/redis/backend` 增加 `healthcheck`
    - `frontend` 与 `celery_worker` 依赖 `backend: service_healthy`
  - 前端错误提示增强：对网络不可达错误（`ERR_NETWORK/ECONNREFUSED`）单独提示“backend 未就绪”，不再只显示笼统兜底文案。
- **验证**
  - `docker compose up -d` 后服务状态显示 backend/postgres/redis 为 `healthy`。
  - 后端日志出现 `POST /api/crawl/tasks` 返回 201，前端点击后可见“已创建采集任务”。
  - 前端近期日志不再出现新的 `http proxy error: /api/crawl/tasks`。

### 问题 3：任务创建成功后采集计数长期为 0

- **现象**
  - 任务可创建，Celery 日志显示任务执行成功，但 `total_count/success_count` 多次为 0。
- **根因**
  - Celery 在 backend/celery 镜像内执行 `scrapy`，镜像最初缺少 `scrapy-playwright/playwright` 与浏览器运行环境，导致任务“可调度但无有效产出”。
- **修复**
  - 在 `backend/requirements.txt` 补齐 `scrapy-playwright`、`playwright`。
  - 在 `docker/backend.Dockerfile` 固化 Playwright 运行库与浏览器安装。
  - 重建 `backend/celery_worker` 并在容器内复验任务链路。
- **验证**
  - 新建任务后状态从 `running` 到 `success`，并出现非 0 计数（如 `total_count=20`）。
  - Mongo 按 `crawl_task_id` 可查到新增记录，PostgreSQL `jobs` 总量同步增长。
