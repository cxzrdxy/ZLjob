# 项目爬虫总体方案（Scrapy + Playwright）

## 1. 方案定位

本方案升级为项目级爬虫总体方案，不再仅覆盖“实时小窗”。  
目标是统一定义采集架构、任务链路、数据流、运维治理和可视化预览能力。

---

## 2. 总体目标

- 建立稳定的职位数据采集主链路。
- 兼容强 JS 页面和动态翻页场景。
- 提供任务可观测能力（状态、日志、预览画面）。
- 保证数据可落库、可去重、可回溯。
- 支持后续扩展多城市、多关键词、多站点。

---

## 3. 技术决策

- 采集框架：Scrapy（调度、解析、清洗、管道入库）。
- 渲染引擎：Playwright（页面加载、执行 JS、翻页、截图）。
- 集成方式：scrapy-playwright 一体化。
- 任务编排：Django API + Celery worker。
- 数据存储：Mongo（采集中间层）+ PostgreSQL（业务查询层）。

---

## 4. 系统架构

## 4.1 核心链路

- 前端发起采集任务。
- 后端创建任务记录并投递 Celery。
- Celery 启动 Scrapy 任务。
- Spider 核心请求默认 `meta["playwright"] = True`，由 Playwright 渲染后返回。
- Spider 解析 Item 并进入 Pipeline。
- Pipeline 写入 Mongo，并同步落 PostgreSQL。
- 前端持续读取任务状态与预览元数据。

## 4.2 职责边界

- Scrapy：请求调度、重试、数据解析、Pipeline。
- Playwright：浏览器行为、JS 渲染、分页控制、页面截图。
- API 层：任务创建、状态查询、预览接口、鉴权控制。
- Celery：异步执行、隔离前端请求时延、失败重试。

---

## 5. 数据流设计

## 5.1 任务流

- `pending -> running -> success/failed`。
- 任务状态与统计字段实时更新（总数、成功数、失败数）。
- 失败任务记录错误摘要和最后步骤。

## 5.2 数据流

- 以 `source_url` 作为幂等主键。
- 去重写入策略：
  - 已存在则更新关键字段与更新时间。
  - 不存在则新增记录。
- 保留原始抓取时间与来源标记，便于审计和问题回溯。

## 5.3 预览流

- Playwright 周期性截图（建议 1~2 秒）。
- 写入预览元数据：
  - `task_id`
  - `latest_frame_url`
  - `latest_step`
  - `latest_at`
  - `error_message`
- 前端小窗轮询 `preview` 接口展示实时画面。

---

## 6. 接口规划

- `POST /api/crawl/tasks`：创建采集任务。
- `GET /api/crawl/tasks`：获取任务列表与状态。
- `GET /api/crawl/tasks/{id}/preview`：获取任务最新预览帧。
- `GET /api/crawl/tasks/{id}/preview/history`：获取预览历史帧（可选）。

---

## 7. 实施清单

## 7.1 爬虫层

- 接入 scrapy-playwright 依赖与 settings。
- Spider 核心请求默认启用 Playwright。
- 配置等待策略、超时、重试和节流。
- 在解析阶段统一输出标准化 Item。

## 7.2 后端层

- 扩展任务模型与预览元数据模型。
- 完善任务状态更新与失败原因落库。
- 提供预览查询接口和鉴权保护。
- 保证 Celery 子进程异常可感知、可回收。

## 7.3 前端层

- Dashboard 显示任务状态统计与列表。
- 增加“实时预览”小窗。
- 显示当前步骤、更新时间、失败文案。
- 任务结束后保留最后一帧。

## 7.4 部署层

- Docker 增加 Playwright 运行时依赖。
- 按机器资源设置并发上限。
- 配置日志采集与容器健康检查。

---

## 8. 参数建议（初始值）

- Playwright 页面并发：2~4。
- Scrapy 并发请求：4~8。
- 页面超时：30~45 秒。
- 失败重试：2~3 次。
- 截图频率：1000~2000ms。
- 任务轮询间隔：2000~4000ms。

---

## 9. 运维与治理

- 监控指标：
  - 任务成功率
  - 平均采集时长
  - 单任务抓取条数
  - 预览接口延迟
- 故障处理：
  - 连续失败阈值告警
  - 超时任务自动终止
  - worker 异常自动拉起
- 成本控制：
  - 并发限流
  - 截图保留时长
  - 过期帧清理策略

---

## 10. 分阶段路线

## 阶段 M1：主链路可用

- 跑通 scrapy-playwright 抓取链路。
- 跑通任务创建、执行、状态查询。
- 跑通基础数据入库。

## 阶段 M2：可视化可用

- 上线预览元数据与截图链路。
- 前端小窗可实时展示任务画面。
- 增加失败提示和任务恢复体验。

## 阶段 M3：稳定性提升

- 增强重试、回退、限流、告警。
- 优化资源消耗和平均任务时长。
- 完成压测和容量评估。

---

## 11. 验收标准

- 任务可稳定执行并完成入库。
- 强 JS 页面可稳定抓到有效数据。
- 小窗可在任务运行期间持续更新。
- 失败任务可定位具体错误并支持重试。
- 连续运行期间系统无明显资源失控。

---

## 12. 结论

- 方案已升级为项目级爬虫总体方案。
- 实时小窗是其中一项可观测能力，而非唯一目标。
- 当前建议按 M1 -> M2 -> M3 推进，先稳主链路再扩体验能力。

## 13. BOSS直聘专项实施（登录 + 滚动）

适用入口：`https://www.zhipin.com/wuhan/?ka=header-home` 及同类搜索/列表页。  
本章节用于指导可执行实现，聚焦两个核心场景：登录态维持与滚动加载。

## 13.1 登录场景实施

## 13.1.1 登录目标

- 在合法授权前提下建立稳定会话。
- 将会话复用于同一采集任务，减少重复登录。
- 登录失败可识别、可回退、可告警。

## 13.1.2 实施步骤

- 启动 Playwright context 前加载历史存储态（如 `storage_state`）。
- 打开入口页并检测是否已登录（通过头像/个人中心元素或登录按钮状态判断）。
- 未登录时进入登录流程：
  - 优先人工可控登录（扫码/人工打码）作为首期方案。
  - 登录成功后立即保存 `storage_state`。
- 采集任务内复用同一 browser context，不重复创建会话。
- context 失效时触发一次“刷新会话”流程；仍失败则任务标记为 `failed` 并写入错误原因。

## 13.1.3 关键检测点

- `is_logged_in`：页面是否处于登录态。
- `login_expired`：请求被重定向登录页或出现未登录提示。
- `session_refresh_result`：会话刷新成功/失败。

## 13.1.4 异常处理

- 登录元素长期未出现：判定页面异常或风控触发，记录截图并中断任务。
- 登录后立即失效：标记会话污染，清理本地状态并要求重登。
- 连续两次会话刷新失败：触发告警并熔断该任务。

## 13.2 滚动场景实施

## 13.2.1 滚动目标

- 稳定触发列表懒加载，获取完整职位卡片数据。
- 控制滚动节奏，避免触发高频行为风控。
- 能判断“已到末尾”并正确停止。

## 13.2.2 实施步骤

- 首屏等待职位列表容器加载完成。
- 执行增量滚动循环：
  - 每次滚动固定像素或滚至倒数第 N 张卡片位置。
  - 每次滚动后等待短暂抖动时间（随机区间）。
  - 读取当前卡片总数与页面高度。
- 触发采集：
  - 对新出现卡片提取摘要信息与详情链接。
  - 详情页按同样 context 打开并解析。
- 终止条件（任一满足即停止）：
  - 连续 K 次滚动卡片数量不再增长。
  - 到达“无更多职位”标识。
  - 达到任务配置上限（页数/条数/时长）。

## 13.2.3 参数初值建议

- 单次滚动间隔：800~1500ms（随机）。
- 连续无增长阈值：3~5 次。
- 单任务最大滚动轮次：50~120。
- 详情页并发打开数：1~2。

## 13.2.4 异常处理

- 滚动后卡片持续为 0：回退到刷新页面并重试一次。
- 滚动触发验证码/风控提示：暂停任务并标记人工介入。
- 页面卡死超时：关闭当前页并重建 tab 后继续。

## 13.3 登录与滚动的协同规则

- 登录成功后再启动滚动；未登录状态禁止进入滚动循环。
- 滚动过程中若检测到登录失效，立即中止滚动并先恢复会话。
- 每轮滚动后写入预览元数据，确保小窗可实时显示当前步骤。

## 13.4 与现有任务模型对齐

- `latest_step` 建议枚举：
  - `boot_context`
  - `check_login`
  - `do_login`
  - `load_list`
  - `scrolling`
  - `parse_detail`
  - `persist_data`
  - `done`
- `error_message` 统一记录短语义错误码，便于聚合统计。

## 14. BOSS专项验收标准

- 登录专项：
  - 新任务可自动识别登录态并进入采集。
  - 会话失效后可自动刷新一次并继续任务。
- 滚动专项：
  - 列表滚动可稳定触发懒加载，抓取数量随轮次增长。
  - 终止条件准确生效，无无限滚动。
- 可观测专项：
  - 小窗可显示登录/滚动阶段状态切换。
  - 异常中断时可看到最后步骤与错误原因。
