1) 之前登录功能修复阶段遇到的问题；

## 一、登录问题修复复盘

### 问题 1：未登录状态访问业务页，体验不稳定

- **现象**
  - 访问首页或职位页时，未登录用户可能无法立即感知需要登录。
- **根因**
  - 业务路由需要统一基于登录状态做前置守卫与跳转策略。
- **修复**
  - 在路由层增加鉴权重定向逻辑：受保护路由未登录时跳转 `/login`，并保留 `redirect` 参数；已登录访问 `/login` 时回首页。
  - 对应实现：`frontend/src/router/index.js` 中 `resolveAuthRedirect` 和 `beforeEach`。
- **验证**
  - 未登录直达 `/` 或 `/jobs`，会被跳转到 `/login`。
  - 登录后会回跳到原始目标路由。

### 问题 2：登录状态在刷新后容易丢失

- **现象**
  - 页面刷新后，前端状态可能回到初始值，导致再次被判定未登录。
- **根因**
  - 登录状态如果只在内存中保存，刷新后无法恢复。
- **修复**
  - 在认证 Store 中将 `accessToken/refreshToken` 持久化到 `localStorage`，并在初始化时回填。
  - 对应实现：`frontend/src/store/auth.js`。
- **验证**
  - 登录后刷新页面，仍可访问受保护页面。

### 问题 3：登录失败提示可用性不足

- **现象**
  - 用户登录失败时很难判断是账号密码错误还是服务不可用。
- **根因**
  - 异常提示信息过于笼统，未优先展示后端返回消息。
- **修复**
  - 登录失败时优先读取后端 `message`，否则回退通用提示。
  - 对应实现：`frontend/src/views/Login.vue` 的 `onSubmit` 异常处理。
- **验证**
  - 后端返回明确报错时，页面可直接提示具体失败原因。

## 二、本次采集“无动静”修复复盘

### 问题 1：点击“开始采集”后用户看不到过程反馈

- **现象**
  - 点击按钮后仅出现“已创建采集任务”，页面缺少任务执行过程与结果展示，主观感受为“没有反应”。
- **根因**
  - 前端只调用创建任务接口，没有任务列表展示，也没有状态轮询。
  - 采集是 Celery 异步执行，创建成功不等于采集完成。
- **修复**
  - 新增任务列表接口调用：
    - `frontend/src/api/index.js` 增加 `fetchCrawlTasks`。
  - Dashboard 新增“采集任务状态”面板与状态标签：
    - `frontend/src/views/Dashboard.vue` 增加任务表格、状态映射、计数展示。
  - 新增自动轮询：
    - 当任务为 `pending/running` 时，按间隔刷新任务列表；
    - 任务全部结束时停止轮询；组件卸载时清理定时器。
- **验证**
  - 创建任务后可看到任务行即时出现；
  - 状态会在“排队中/采集中/已完成/失败”之间变化；
  - 成功/失败计数会更新。

### 问题 2：本地直接构建受 PowerShell 执行策略影响

- **现象**
  - 在 Windows 本地终端执行 `npm run build` 报脚本执行策略错误。
- **根因**
  - 本地 PowerShell 执行策略限制 `npm.ps1`。
- **修复**
  - 使用容器内命令进行构建验证，避免宿主机策略差异干扰：
    - `docker compose exec frontend npm run build`
- **验证**
  - 容器内构建通过。

### 问题 3：任务已执行但抓取结果为 0（非本次 UI 反馈问题）

- **现象**
  - Celery 日志显示任务执行成功，但 `total_count` 为 0。
- **根因**
  - 说明“任务触发链路”正常，但爬虫侧数据抓取结果为空，常见原因包括目标站反爬、页面结构变化、请求参数不匹配等。
- **处理结论**
  - 本次已先修复“无反馈”问题；
  - 抓取结果为 0 属于下一阶段“爬虫有效数据率”问题，需单独排查 spider 与目标站兼容性。

## 三、这次沉淀下来的排查准则

- 异步任务问题优先拆成两层看：  
  1) **是否触发成功**（API -> Celery）；  
  2) **是否执行有效**（Spider -> 数据入库）。
- 任何“按钮无反应”类问题，优先补“可观察性”：  
  - 状态展示、轮询刷新、失败提示。
- 验证优先使用与部署一致的运行环境：  
  - 本项目优先使用 Docker 容器内执行构建和日志排查。

## 五、本次“环境固化到镜像”复盘

### 问题 1：直接 `docker compose build crawler` 构建失败

- **现象**
  - 构建阶段拉取 `python:3.11-slim` 失败，报 Docker Hub 连接超时。
- **根因**
  - 当前环境到 `registry-1.docker.io` 网络不稳定，基础镜像元数据获取失败。
- **修复**
  - Dockerfile 增加 `ARG BASE_IMAGE`，默认仍为 `python:3.11-slim`，并支持在网络受限时切换到本地已存在镜像。
  - 使用本地已有 `boss-job-platform-crawler:latest` 作为基底完成一次增量重建，确保固化流程可落地。
- **验证**
  - 镜像成功构建并替换运行中的 crawler 容器。

### 问题 2：容器内缺少 `scrapy-playwright` 导致启动报错

- **现象**
  - 爬虫启动时报 `ModuleNotFoundError: No module named 'scrapy_playwright'`。
- **根因**
  - 运行容器基于旧镜像，镜像内依赖未更新到最新 requirements。
- **修复**
  - 先在容器内临时安装依赖恢复可用性；
  - 再将依赖安装步骤固化到 Dockerfile，并重建镜像保证后续环境一致。
- **验证**
  - 容器内 `scrapy check` 可通过，`scrapy crawl` 可启动到 Playwright 阶段。

### 问题 3：Playwright 配置路径不兼容

- **现象**
  - 启动时报 `No module named 'scrapy_playwright.middleware'`。
- **根因**
  - `scrapy-playwright` 新版本不再提供该 middleware 路径，错误配置导致中间件加载失败。
- **修复**
  - 移除 `DOWNLOADER_MIDDLEWARES` 中无效项，仅保留 `DOWNLOAD_HANDLERS` 的 Playwright 处理器配置。
- **验证**
  - Scrapy 可正常加载下载器并启动 spider。

### 问题 4：浏览器二进制与系统依赖不完整

- **现象**
  - 报错 `Executable doesn't exist ... headless_shell`，或提示缺少 `libglib/libnss/libx11` 等运行库。
- **根因**
  - 镜像中未完整安装 Playwright 运行依赖与 headless shell 浏览器。
- **修复**
  - 在 Dockerfile 固化安装系统库与字体；
  - 固化 `playwright install --only-shell chromium`；
  - 固化 `PLAYWRIGHT_BROWSERS_PATH=/ms-playwright`，确保浏览器路径可预测。
- **验证**
  - 容器内执行 Playwright 启动脚本输出 `playwright_ok`；
  - 采集命令日志出现 `Browser chromium launched`。

### 问题 5：容器内 apt 安装偶发锁冲突

- **现象**
  - 安装依赖时出现 `Could not get lock /var/lib/dpkg/lock-frontend`。
- **根因**
  - 前一次 apt 进程未完全释放锁文件，导致后续安装失败。
- **修复**
  - 清理锁并执行 `dpkg --configure -a` 后重试安装流程。
- **验证**
  - 系统库安装成功，后续 Playwright 安装和启动均正常。
